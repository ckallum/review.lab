review.dev · system architecture

A local PR review surface, alongside GitHub.

Claude Code's /publish-review skill shells out to reviewdev publish, which uploads the current branch's diff to a per-repo reviewdev serve instance. Every successful publish creates an immutable revision with its own hunks, chapters, decisions, and comments. The browser opens to a story-shaped view of the latest revision; a diff view links each revision to its predecessor. GitHub remains canonical.

Draft 4 May 2026 Single-user Local-first No telemetry Bun · Hono · SQLite

§ 02 · Processes

Three runtimes, one shared store

The publish CLI is one-shot. The serve process is long-running, per-repo, foreground. The browser is stateless and reconnects on demand.

One-shot

reviewdev publish

Called by the Claude Code skill. Resolves the diff vs origin/<base> after git fetch + merge-base, hashes hunks, uploads via POST /api/pr, opens the browser.

  • Reads $CLAUDE_SESSION_ID + transcripts
  • Best-effort gh pr view --json url
  • Returns URL within 2s (p95)
lives ~5s

Long-running

reviewdev serve

Per-repo Hono server. Probes ports 7891–7899, writes the chosen one to <repo>/.reviewdev/port. Runs migrations on startup. Streams chapter generation via SSE.

  • HTTP + SSE; binds to localhost only
  • Writes via BEGIN IMMEDIATE
  • Foreground process (tmux / launchd)
lives indefinitely

Stateless

Browser · Concept 01

Renders web/index.html data-driven from /api/pr/:id/rev/:n. Skeleton paints immediately; chapters stream in over EventSource. Old tabs pin to the revision they opened on.

  • Vanilla JS — no build step
  • Reconnects on connection drop
  • No revision auto-refresh
lives per-tab
§ 03 · Data model

Ten tables, scoped by revision

One SQLite DB per repo at <repo-root>/.reviewdev/db.sqlite. WAL mode; numbered SQL migrations applied on serve start. Revisions are immutable — every publish snaps a new one.

Table Scope What it holds
pulls Identity One per branch. Title, base, github_url, status.
sessions Identity Claude Code sessions correlated to this branch via cwd. Sessions span revisions.
revisions Revision number, git_head_sha, git_base_sha, diff_hash. Duplicate-publish detection via diff_hash.
hunks Revision Diff hunks. id = SHA-256(file_path + content). Same hash can appear in multiple revisions.
chapters Revision LLM-generated groupings. inherited_from_chapter_id tracks reuse from the prior revision.
chapter_hunks Revision Many-to-many within a revision.
decisions Revision Extracted in the same LLM call as chapters.
comments User action Markdown body. Scoped to the revision they were left on. No migration.
approvals User action Chapter-level signoff on a specific revision.
usage Observability One row per LLM call. Drives the $5 soft warning and REVIEWDEV_DAILY_CAP hard cap.

§ 05 · Revision diff

What you actually changed

GET /pr/:id/rev/:n/diff renders code delta + chapter delta against revision n−1. The link between two immutable snapshots.

no prior to diff] B -->|no| D[Load rev n hunks + chapters
and rev n-1 hunks + chapters] D --> E[Diff hunks by content-hash] D --> F[Diff chapters by title

+ sha:7] H --> I

Hunk delta is pure set comparison on content hashes. Chapter delta is title + membership + the inherited_from_chapter_id back-pointer. No comment delta — comments are revision-scoped by design.

§ 06 · Concurrency

Two terminals, two revisions

Immutable revisions make concurrent publishes trivially safe. No supersedes, no cancellation, no shared mutable state.

sequenceDiagram actor T1 as Terminal A actor T2 as Terminal B participant Server as serve participant DB as SQLite participant LLM as Anthropic T1->>Server: POST /api/pr (diff A) Server->>DB: BEGIN IMMEDIATE Server->>DB: INSERT revision N+1 Server-->>T1: rev N+1 URL T2->>Server: POST /api/pr (diff B) Server->>DB: BEGIN IMMEDIATE Server->>DB: INSERT revision N+2 Server-->>T2: rev N+2 URL Note over Server,LLM: SSE streams run independently per revision T1->>Server: GET SSE rev N+1 generate T2->>Server: GET SSE rev N+2 generate Server->>LLM: prompt for rev N+1 Server->>LLM: prompt for rev N+2 LLM-->>Server: chapters rev N+1 LLM-->>Server: chapters rev N+2 Server-->>T1: events rev N+1 Server-->>T2: events rev N+2

BEGIN IMMEDIATE serialises the inserts. Both publishes succeed; whichever commits first is N+1, the other N+2.

§ 07 · Load-bearing decisions

The dozen choices that shape everything else

Full Key Decisions table lives in design.md. These are the ones that, if reversed, would touch the rest of the system.

  • Revisions are immutableno migration

    Each publish snaps a new revisions row. Comments live on their revision. The diff view is the link. Dropped four hard correctness surfaces (hash-based comment migration, locked chapters, orphan handling, SSE supersedes).

  • Per-repo SQLite, not global<repo>/.reviewdev/db.sqlite

    No P0 feature uses cross-repo. Simpler isolation. Routing becomes trivial — the cwd is the routing key.

  • Single combined LLM call{chapters, decisions}

    Chapters and decisions emerge from one call instead of two parallel. ~40% cost saving, ~30% latency saving, no consistency drift.

  • Chapter inheritance via prompt hintsoft, not locked

    Prior chapter titles + surviving hunk hashes get passed in. Model decides reuse. No lock UI, no orphan handling, no recompose button.

  • Hunk identity is order-sensitiveSHA-256(path + content)

    Reordered lines deliberately register as add/remove. Refactors are meaningful changes and belong in the diff view.

  • Foreground reviewdev serveno daemon

    User runs it in tmux or launchd. No PID files, no auto-start, no surprise lifecycle. publish errors fast if no server is up.

  • SSE for chapter deliveryEventSource

    Time-to-URL stays sub-2s; chapters paint as they arrive. Stale tabs pin to their revision — no auto-supersede.

  • Confidence = high for week 1column reserved

    Heuristic was on the cut-list. Schema column exists; values become real if dogfood reveals the gap.

  • Cost guardrails in DB, not vendor billingusage table

    $5/day soft warning, REVIEWDEV_DAILY_CAP hard cap. One env var, one row per call. No third-party billing dependency.

  • Demo HTML is the source of truth for the viewweb/index.html

    Concept 01 extracted from the reference demo, parameterised to fetch from the API. Concepts 02–06 dropped during import.

§ 08 · Failure modes

What can go wrong, and what shows up

Every failure path has at least one of: a test, an error path, or a non-silent user signal. Full table in SPEC.md §Failure modes.

External · API
Anthropic rate-limit / 5xx
Retry with backoff, then surface a banner on the chapter pane: "chapter generation failed — retry".
External · git
Stale or unreachable origin
10s fetch timeout, continue with last-known base. Warning line: "fetch failed, using last-known base".
External · gh
gh missing or unauthenticated
stderr swallowed silently. Header shows branch name instead of "View on GitHub" link.
Local · transcript
Session file missing or /compacted
Continue without transcript context. Compacted sessions surface a "compacted" badge in the session bay.
Local · port
All of 7891–7899 taken
Fail with: "no free port in 7891–7899, set REVIEWDEV_PORT".
Local · cost
REVIEWDEV_DAILY_CAP reached
Publish fails before the LLM call. Clear message: "Daily cap reached, override via REVIEWDEV_DAILY_CAP=N".
User · re-publish
Same diff republished
diff_hash matches latest revision. Same URL returned. No new revision created. No surprise.
User · concurrent
Two terminals publish same branch
Both succeed. Revisions assigned in commit order via BEGIN IMMEDIATE. Both readable.
User · stale tab
Tab from yesterday's revision still open
Keeps showing that revision. Reload → latest. Explicit, deterministic — by design.
